## 第五部分：平台2功能详细设计

平台2是企业知识沉淀和AI能力进化的“大脑”，其核心用户是懂业务但不懂技术的企业产品经理（如周七），以及提供技术支持的技术专家（如孙八）。本部分的设计核心是“低代码/无代码”，让业务专家可以像“搭积木”一样，将自己的业务思想构建成平台1中可用的AI能力。

### 5.1. 价值领域配置器 (Value Domain Configurator)

这是平台2最核心的模块，它提供了一套“CUI+GUI”的混合界面，引导产品经理完成从业务流程梳理到AI能力配置的全过程。

#### 5.1.1. 功能：价值领域管理

**1. 功能说明**

允许用户创建、编辑和管理价值领域。一个价值领域对应企业的一项核心业务活动，如“需求洞察”、“方案设计”、“资源采购”。

**2. 用户故事**

- **作为一名企业产品经理**，我想要**在一个地方看到公司所有已配置的价值领域**，以便**进行统一的管理和维护**。
- **作为一名企业产品经理**，我想要**能够创建一个新的价值领域**，并**定义它的目标、边界和核心阶段**，以便**将一项新的业务活动纳入平台的管理范围**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.1.1.1 | 价值领域列表与仪表盘 | P0 |
| 5.1.1.2 | 创建新的价值领域 | P0 |
| 5.1.1.3 | 编辑价值领域基本信息（定义、边界、价值） | P0 |
| 5.1.1.4 | 复制/克隆一个已有的价值领域 | P1 |
| 5.1.1.5 | 归档或禁用价值领域 | P1 |
| 5.1.1.6 | 查看价值领域的使用统计（调用次数、成功率） | P2 |

**4. 交互流程**

- **主界面**: 平台2首页即为价值领域仪表盘，以卡片形式展示所有价值领域。
- **卡片内容**: 领域名称、一句话描述、负责人、状态（已发布/草稿）、最近更新时间、核心指标（如关联的Skill数量、月调用次数）。
- **创建流程**: 
  1. 点击“创建价值领域”按钮。
  2. 弹出对话框，引导用户输入：“你想要创建一个什么样的价值领域？请用一句话描述它的核心业务活动。”（例如：“帮助客户代表进行售前的客户需求分析和洞察”）
  3. AI根据用户输入，自动推荐领域名称、定义和边界，并生成草稿。
  4. 用户进入新创建的价值领域配置页面，开始进行业务阶段划分等后续配置。

**5. 界面设计 (原型描述)**

- **仪表盘**: 采用现代化的卡片式布局，提供搜索和筛选功能。
- **创建引导**: CUI引导式创建，降低用户的心理门槛。AI的自动填充能力是关键。

**6. 数据结构**

**ValueDomain (价值领域) 数据表**

| 字段名 | 数据类型 | 描述 | 示例 |
| :--- | :--- | :--- | :--- |
| `domain_id` | String | 唯一标识符 (主键) | `domain_demand_insight` |
| `name` | String | 领域名称 | "需求洞察" |
| `definition` | Text | 详细定义 | "在售前阶段..." |
| `boundary` | Text | 业务边界 | "从接触客户开始，到..." |
| `value_proposition` | Text | 核心价值 | "提升需求分析的..." |
| `owner_id` | String | 负责人用户ID | `user_zhouqi` |
| `status` | Enum | 状态 | `DRAFT`, `PUBLISHED`, `ARCHIVED` |

#### 5.1.2. 功能：业务流程编排

**1. 功能说明**

允许产品经理在一个可视化的界面上，通过拖拽和连接的方式，定义一个价值领域下的标准业务流程。

**2. 用户故事**

- **作为一名企业产品经理**，我想要**像画流程图一样，定义出‘需求洞察’的5个步骤**，并且**为每个步骤指定需要使用的工具和产出物**，以便**将我脑中的业务流程固化下来**。
- **作为一名企业产品经理**，我想要**能够为同一个价值领域，根据不同行业（如金融、制造）配置不同的业务流程分支**，以便**提供更具行业特色的智能引导**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.1.2.1 | 可视化流程编排画布 | P0 |
| 5.1.2.2 | 业务阶段/步骤节点的拖拽与创建 | P0 |
| 5.1.2.3 | 节点间依赖关系的连接 | P0 |
| 5.1.2.4 | 为节点配置详细信息（名称、描述、工具、产出物） | P0 |
| 5.1.2.5 | 支持条件分支（如按客户类型、项目金额） | P1 |
| 5.1.2.6 | 流程版本管理 | P1 |
| 5.1.2.7 | AI辅助生成流程草稿 | P1 |

**4. 交互流程**

- **界面**: 左侧为节点工具箱（如“阶段”、“判断”、“并行”），右侧为配置面板，中间为画布。
- **操作**: 用户从工具箱拖拽一个“阶段”节点到画布上，点击节点，右侧配置面板显示该节点的属性表单。
- **配置**: 在配置面板中，用户可以填写阶段名称、描述，并通过下拉框选择该阶段需要使用的工具（如“联网搜索”、“文档分析”），定义产出物（如“客户画像报告.docx”）。
- **AI辅助**: 用户也可以在CUI中输入：“‘需求洞察’分为5个阶段：客户画像分析、需求挖掘...”，AI将自动在画布上生成对应的5个节点和初步连线。

**5. 技术实现说明**

- 可视化流程编排器可基于开源库如`React Flow`或`AntV X6`进行开发。
- 画布上的每一次操作（创建节点、连接）都应实时保存，并触发后台的本体数据更新。

### 5.2. 本体编辑器 (Ontology Editor)

本体编辑器是为高级用户准备的“专家模式”。大多数情况下，产品经理在价值领域配置器中操作，本体数据会自动生成。但当需要精细化调整或处理复杂逻辑时，可以进入本体编辑器直接操作底层的知识图谱。

#### 5.2.1. 功能：知识图谱可视化

**1. 功能说明**

以图形化的方式展示本体数据，让用户可以直观地看到对象、属性和它们之间的链接关系。

**2. 用户故事**

- **作为一名技术专家**，我想要**查看‘需求洞察’这个业务流程自动生成的本体结构**，以便**我能确认其逻辑是否正确，以及是否需要优化**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.2.1.1 | 知识图谱的可视化渲染 | P1 |
| 5.2.1.2 | 节点的展开与折叠 | P1 |
| 5.2.1.3 | 搜索与高亮特定对象或关系 | P1 |
| 5.2.1.4 | 查看节点（对象）的详细属性 | P1 |

**4. 技术实现说明**

- 可采用`D3.js`, `vis.js`或`ECharts`的图关系图来实现知识图谱的可视化。
- 后端需提供GraphQL或Cypher查询接口，供前端获取图数据。

### 5.3. Skill与工具管理器 (Skill & Tool Manager)

这里是平台所有“能力”的注册和管理中心。

#### 5.3.1. 功能：Skill管理

**1. 功能说明**

管理由本体数据自动转换而来的动态Skill，包括查看、版本控制和测试。

**2. 用户故事**

- **作为一名企业产品经理**，我想要**在我完成一个业务流程配置后，系统能自动生成一个对应的Skill**，并且**我能在一个测试环境中模拟运行它**，以便**在发布前验证其效果**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.3.1.1 | 自动从本体配置生成Skill模板 | P1 |
| 5.3.1.2 | Skill列表与版本管理 | P1 |
| 5.3.1.3 | Skill代码查看器（只读） | P2 |
| 5.3.1.4 | Skill模拟运行与调试环境 | P1 |

**4. 交互流程**

- 在价值领域配置页面，当产品经理点击“发布”时，系统后台触发Skill生成流程。
- 生成成功后，在Skill管理器中会出现一个新的Skill条目，状态为“草稿”。
- 产品经理可以点击“测试”，进入一个模拟的CUI环境，与这个新生成的Skill进行对话，验证其流程和逻辑是否正确。
- 测试通过后，可将Skill发布，平台1的用户便可在经验市场中看到并使用。

---

#### 5.3.2. 功能：MCP工具注册

**1. 功能说明**

允许技术专家将企业内部的系统或第三方服务，封装成标准的MCP（Model Context Protocol）工具，注册到平台中供业务流程调用。

**2. 用户故事**

- **作为一名技术专家**，我想要**将公司的CRM系统查询客户信息的接口，注册成一个名为‘查询客户信息’的工具**，以便**产品经理在配置业务流程时可以直接使用它**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.3.2.1 | MCP工具注册表单 | P1 |
| 5.3.2.2 | 工具的API端点、认证方式、输入输出参数定义 | P1 |
| 5.3.2.3 | 工具的在线测试功能 | P1 |
| 5.3.2.4 | 工具的使用文档和示例自动生成 | P2 |

**4. 技术实现说明**

- MCP工具的定义需遵循OpenAPI (Swagger) 规范。
- 平台提供一个GUI表单，引导技术专家填写API的各项描述，后台自动生成符合规范的JSON或YAML文件。

### 5.4. 经验审核中心 (Experience Audit Center)

#### 5.4.1. 功能：审核工作台

**1. 功能说明**

为平台2的产品经理提供一个集中的工作台，审核来自平台1用户提交的“经验发布”申请。

**2. 用户故事**

- **作为一名企业产品经理**，我想要**在一个队列中看到所有待我审核的经验**，并且**能清晰地看到经验的来源（谁、在哪个项目中沉淀的）、内容和建议的复用场景**，以便**我能高效地做出审核决策**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.4.1.1 | 待审核经验队列 | P1 |
| 5.4.1.2 | 经验详情审查视图 | P1 |
| 5.4.1.3 | 审核操作（通过、拒绝、打回修改） | P1 |
| 5.4.1.4 | 审核意见填写 | P1 |
| 5.4.1.5 | 审核历史记录 | P2 |

**4. 交互流程**

- **详情视图**: 点击一个待审核项，进入详情页。页面左侧展示该经验的完整内容（如一段巧妙的分析过程、一个好用的文档模板），右侧为审核信息面板。
- **审核信息**: 面板中包含提交人、来源项目、AI自动生成的经验总结、建议的适用范围等。
- **操作**: 产品经理可以在面板下方填写审核意见，并点击“通过”或“拒绝”按钮。如果选择“通过”，则需要进一步完善该经验的元数据（如名称、标签、简介），然后正式发布到平台经验市场。

### 5.5. 分析与洞察仪表盘 (Analytics & Insights Dashboard)

#### 5.5.1. 功能：数据可视化

**1. 功能说明**

通过可视化的图表，向管理者展示平台的整体运营情况、知识沉淀与复用的效果，以及为业务带来的价值。

**2. 用户故事**

- **作为一名管理者**，我想要**看到一个仪表盘，告诉我本月沉淀了多少条经验、被复用了多少次、为公司节省了多少工时**，以便**我能直观地了解这个平台的ROI**。

**3. 功能清单**

| 编号 | 功能点 | 优先级 |
| :--- | :--- | :--- |
| 5.5.1.1 | 平台核心运营指标（用户活跃度、项目数） | P2 |
| 5.5.1.2 | 经验沉淀与复用漏斗图 | P2 |
| 5.5.1.3 | Top 10最受欢迎的经验/Skill榜单 | P2 |
| 5.5.1.4 | 效率/质量提升趋势图（如项目平均周期、方案一次通过率） | P2 |
| 5.5.1.5 | 可自定义的报表与下钻分析 | P2 |

**4. 技术实现说明**

- 可采用成熟的BI工具（如Metabase, Superset）或前端图表库（ECharts）进行开发。
- 后端需要建立一个数据仓库或数据集市，对业务数据进行ETL处理，以支持分析查询。
